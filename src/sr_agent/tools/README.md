# Writing Tools

This directory contains tools that can be called by `SRAgent` through an LLM
tool parser. A new tool should inherit from `BaseTool`, register itself with a
stable tool name, and implement `execute()`.

## Minimal Tool

```python
from typing import Any, Dict

from .base_tool import BaseTool, ToolMetadata


@BaseTool.register("my_tool")
class MyTool(BaseTool):
    metadata = ToolMetadata(name="my_tool")

    def execute(self, value: str, limit: int = 10) -> Dict[str, Any]:
        """Short description of what this tool does.

        Args:
            value: Input value to process.
            limit: Maximum number of items to return.

        Returns:
            Result dictionary.
        """
        return {"value": value, "limit": limit}
```

## Required Pieces

- Inherit from `BaseTool`.
- Add `@BaseTool.register("tool_name")` so the tool can be discovered.
- Set `metadata = ToolMetadata(name="tool_name")`.
- Implement `execute()`. Its arguments are exposed to the LLM, so keep them
  simple and serializable.

`BaseTool` automatically uses `execute()` to infer missing metadata:

- `metadata.description` is inferred from the beginning of the `execute()`
  docstring, stopping before `Args:`.
- `metadata.parameters` is inferred from the `execute()` signature, type hints,
  defaults, and `Args:` descriptions.

Class docstrings are useful for humans reading the code, but they are not used
to construct tool descriptions.

## Recommended Metadata Style

Prefer writing a clear Google-style docstring on `execute()`. This keeps the
runtime behavior, LLM-facing description, and parameter documentation in one
place.

Write `ToolMetadata(description=..., parameters=...)` manually when automatic
inference is not precise enough, for example when parameters need stricter JSON
Schema constraints, enum explanations, nested objects, or special formats.
Manual metadata is not overwritten by `BaseTool`.

## Context

Runtime context is passed when the tool is instantiated and is available at
`self.context`. `SRAgent.fit()` currently instantiates tools with:

```python
{"data": X | y, "target": target_name}
```

Use context for data, model handles, caches, or other values that should not be
generated by the LLM as `execute()` arguments.

## Return Values

`execute()` should return a result dictionary.

`BaseTool.__call__()` wraps the output in `ToolCallResult`, catches unexpected
exceptions, records basic metadata, and calls `format_result_dict()` to create
the string shown back to the LLM.

Override `format_result_dict(cls, result)` when the raw result dictionary is too
large or not readable enough for the LLM.

## Formula And Metrics Contract

If a tool accepts, evaluates, fits, transforms, or produces a symbolic
regression formula, return `formula` and `metrics` fields whenever possible:

```python
return {
    "formula": "2.0 * x1 + 3.0",
    "metrics": {
        "mse": 0.001,
        "rmse": 0.0316,
        "mae": 0.02,
        "r2": 0.998,
    },
}
```

This is required for integration with `SRAgent.fit()`:

- `update_buffer()` uses `result.get("formula")` and `result["metrics"]["mse"]`
  to select promising LLM branches.
- `update_topk()` only records results whose `result["metrics"]` is a dict and
  contains a non-`None` `mse`.
- If `formula` is missing, `update_topk()` may fall back to the tool call's
  `eq` parameter, but tools that produce a new or fitted formula should return
  the final formula explicitly.

For failed formula-related calls, return a result that makes the failure clear
and does not look like a valid candidate, for example:

```python
return {
    "error": "Could not parse formula",
    "metrics": None,
}
```
