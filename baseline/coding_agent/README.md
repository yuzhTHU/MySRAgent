# Coding-Agent Symbolic Regression Baseline

This directory contains the protocol for testing a coding agent such as Codex or Claude Code on one symbolic-regression problem.

Your task is to discover a simple formula for the target variable from numeric data. Do not inspect benchmark source files, raw data files, metadata files, or any ground-truth expression. Use only the public problem files generated in `/tmp`, the initial result JSON generated by `fetch_problem.py`, and the tool scripts documented below.

Do not start or delegate to sub-agents. This benchmark measures the current coding agent as a single solver, so all exploration, tool calls, result writing, and final evaluation must be performed in this same agent session.

## 1. Fetch One Problem

The scripts automatically switch their working directory to the repository root and add it to `sys.path`, so they can be launched from any directory. Example:

```bash
python baseline/coding_agent/fetch_problem.py \
  --dataset lsrtransform \
  --problem-name II.6.15b_1_0 \
  --agent codex
```

The script prints a JSON manifest and creates these files:

```text
/tmp/sr_agent_coding_agent/<dataset>_<problem-name>/problem.npz
/tmp/sr_agent_coding_agent/<dataset>_<problem-name>/context.npz
/tmp/sr_agent_coding_agent/<dataset>_<problem-name>/manifest.json
logs/baseline/coding_agent/<dataset>_<problem-name>_<agent>.json
```

Important files:

- `problem.npz`: public benchmark data and problem description.
- `context.npz`: input file used by the provided tools.
- `manifest.json`: paths needed for this run.
- initial result JSON under `logs/baseline/coding_agent/`: fill this file with your final expression.

`problem.npz` contains:

- `problem_description`: natural-language problem information.
- `symbols`: variable names. The first symbol is the target; the rest are features.
- `symbol_descs`, `symbol_properties`: variable descriptions and roles.
- `train_X`, `train_y`: training arrays.
- `context_path`: path to `context.npz`.
- `result_path`: path to the initial result JSON.

## 2. Explore The Data

You may write temporary analysis scripts under `/tmp` or another scratch location. You may inspect `problem.npz`. Use `context.npz` only as the `--context` argument to the provided tools.

Useful Python loading snippet:

```python
import numpy as np

problem = np.load("/tmp/sr_agent_coding_agent/.../problem.npz", allow_pickle=True)
symbols = problem["symbols"].tolist()
target = symbols[0]
features = symbols[1:]
X = problem["train_X"]
y = problem["train_y"]
```

## 3. Use Existing Tools

Before writing custom analysis code, you must inspect the available tool interface. The `schema` command prints the accepted JSON parameters for a tool. Run these commands immediately after fetching the problem:

```bash
python baseline/coding_agent/call_tool.py list
python baseline/coding_agent/call_tool.py schema statistics_analysis
python baseline/coding_agent/call_tool.py schema evaluate_formula
python baseline/coding_agent/call_tool.py schema polynomial_fit
```

During the search you must call `evaluate_formula` for every serious candidate before final submission. You should also call at least one structure-discovery tool, such as `polynomial_fit`, `call_sindy`, or `call_pysr`, unless the relationship is already obvious from the required tool outputs. Custom Python analysis is allowed, but it is supplementary; do not skip the tool calls.

Call tools through the wrapper:

```bash
python baseline/coding_agent/call_tool.py list
python baseline/coding_agent/call_tool.py schema evaluate_formula
python baseline/coding_agent/call_tool.py call statistics_analysis \
  --context /tmp/sr_agent_coding_agent/.../context.npz \
  --params '{"variables": ["x1", "x2"]}'
python baseline/coding_agent/call_tool.py call evaluate_formula \
  --context /tmp/sr_agent_coding_agent/.../context.npz \
  --params '{"f": "sin(x1) + x2", "fit": false}'
```

Recommended tools:

- `statistics_analysis`: inspect variable ranges and transformed expressions.
- `evaluate_formula`: score a candidate formula on training data.
- `polynomial_fit`: fit polynomial or engineered-feature models.
- `call_sindy`: try sparse polynomial/trigonometric libraries.
- `call_pysr`: run genetic-programming symbolic regression if dependencies are available.

Tool calls should be used for exploration only. The final score is computed by `evaluate_result.py` on benchmark test data.

## 4. Fill The Result JSON

`fetch_problem.py` already creates a result JSON. Use the `result_path` printed by `fetch_problem.py` or stored in `manifest.json`. When you believe you have found the target formula, update that JSON file.

Fill at least these fields:

```json
{
  "end_time": "2026-05-31T12:15:00+08:00",
  "duration_seconds": 900.0,
  "discovered_expression": "sin(x1) + x2",
  "notes": "short optional notes"
}
```

Keep the existing `agent`, `dataset_identifier`, `equation_id`, `problem_path`, `context_path`, and `start_time` fields.

## 5. Evaluate And Update The Result

Before evaluation, record token usage if your coding-agent environment exposes it. Use the official usage/status display from the agent UI or CLI session, not an estimate. If no exact token count is available, do not invent one; pass `--total-tokens 0` and note that token usage was unavailable in the result `notes`.

Run:

```bash
python baseline/coding_agent/evaluate_result.py \
  --result <result_path_from_manifest> \
  --agent-model "<your model name>" \
  --total-tokens 0
```

The evaluator updates the same JSON file with:

- `id_metrics`: MSE, normalized MSE, R2, Kendall tau, MAPE, Acc@0.1, and valid point count.
- `symbolic_acc`: whether the discovered expression is equivalent to ground truth.
- `symbolic_acc_detail`: reason from the symbolic/numeric equivalence check.
- `gt_expression`: ground-truth formula, added only after final evaluation.
- additional run metadata such as model name and token count.

The result JSON is made read-only after evaluation. Do not try to modify it after `gt_expression` has been written.

Do not repeatedly use final evaluation as an oracle while searching.
